# Self-test run — 2026-04-11 #2 (Loops 2 + 3 combined)

Loops 2 and 3 ran back-to-back. Loop 2 surfaced a critical scoring bug,
which was fixed and verified in Loop 3.

## What I fixed this loop (no-brainers)

- `src/lib/ai/score.ts:69-85` — **CRITICAL**: `renderTimedTranscript` was
  silently replacing the full transcript with ONLY the words that had
  timing data. If Deepgram returned fewer word timings than total words
  in the transcript (common with punctuation, contractions, numbers), or
  if word timings were partial for any reason, Claude would score a
  truncated version of the rep. Fix: always send the full transcript as
  canonical, append a separate "TIMESTAMP INDEX" section for Claude to
  use in callout ranges.

  Before fix: structured 130-word rep → composite **54**, structure **22**
  After fix: same rep → composite **89**, structure **91**

  This would have affected every real user.

- `src/app/api/score/route.ts:198-213` — Zod errors now return a clean
  human-readable `message` (first error's path + message) with raw Zod
  `details` array in a separate key.

## Small-but-judgment (Max to pick)

### J4 — Rate limiting is no-op without Upstash

Not a bug, but worth calling out: all 4 expensive endpoints (score,
transcribe, upload, talking-points) have rate-limit wiring that silently
degrades to "allow everything" when Upstash isn't configured. Before
deploying to a public URL, need either:
- Upstash Redis (free tier: 10k commands/day, takes 2 minutes to set up)
- Or: a local in-memory rate limiter as fallback (Map + sliding window)
